# 란도랜드2 작업 지침

최근 기준: 2026-07-14

이 문서는 저장소 전체에 적용되는 현재 작업 기준이다. 세부 규칙은 `docs/` 문서를 함께 확인하되, 사용자 최신 지시가 문서보다 우선한다.

## 1. 제품과 문구

- 사용자에게 보이는 서비스명은 `란도랜드2`다. `RANDOLAND` 표기는 꼭 필요한 경우가 아니면 사용하지 않는다.
- 사용자 화면은 자연스러운 한국어를 우선한다. 티커, RP, AI처럼 의미가 명확한 용어는 유지할 수 있다.
- 불필요하게 긴 설명, 기능 자랑, 중복 안내를 넣지 않는다. 핵심 행동과 결과를 짧고 명확하게 안내한다.
- 서비스 성격은 `모의 주식시장 리그`로 표현한다.
- 참가 안내 문구는 `리그 참가 시 1,000,000 RP가 지급됩니다.`를 사용한다.
- 이모지는 사용하지 않는다. 동일한 스타일의 아이콘이나 프로젝트 전용 에셋을 사용한다.
- `README.md`는 GitHub에서 참가자들이 보는 소개 문서다. 코드, 기술 스택, 환경변수, 빌드, 배포, 데이터베이스 작업 설명을 넣지 않는다.
- 불필요한 잡다한 설명을 넣지 않는다.

## 2. 현재 운영 상태

- 현재는 한 달짜리 실제 운영 리그가 아니라 데모 리그 하나와 기본 종목 10개를 사용한다.
- 데모 리그와 기본 10종목은 정상 운영 데이터이며 테스트 데이터로 취급해 삭제하지 않는다.
- 화면에 가짜 종목, 임시 주문, 하드코딩된 자산, 미리보기 순위를 만들지 않는다.
- 활성 데이터가 없으면 실제 빈 상태나 운영 준비 상태를 표시한다. 프런트엔드 임시 데이터로 채우지 않는다.
- 별도 요청 없이 추가 리그를 만들거나 현재 데모 리그 일정과 데이터를 바꾸지 않는다.

## 3. 핵심 게임 규칙

- 카카오 계정 하나는 한 리그에 한 번만 참가할 수 있다.
- 참가 시 1,000,000 RP와 행동구 28개·동물 41개 조합의 익명 닉네임을 지급한다.
- 익명 닉네임은 같은 리그에서 중복되지 않아야 하며 참가자가 고르거나 수정하지 않는다.
- 참가자는 리그당 종목을 최대 하나 상장할 수 있고, 자신의 상장 종목은 거래할 수 없다.
- 한 라운드는 하루이며 경계와 모든 운영 시각은 KST 오전 9시다.
- 라운드 중 주문은 대기 상태로 접수한다. 정산 잠금 전에는 자유롭게 취소할 수 있다.
- 매일 오전 9시에는 대기 주문과 만기 청산을 먼저 **변동 전 가격**으로 체결한 후 뉴스와 신규 가격을 공개한다.
- 라운드당 주문 접수 한도는 매수 5건, 매도 5건이다. 취소를 이용해 한도를 우회할 수 없게 한다.
- 매매 1회는 매수 후 다른 라운드에서 전량 매도해 포지션이 0이 된 한 사이클이다.
- 존버기간은 포지션을 유지한 완료 라운드 수로 계산한다.
- 주간 순위는 매주 일요일 오전 9시 정산 후에만 갱신해 공개한다. 그 사이의 실시간 순위는 노출하지 않는다.
- 리그 종료 입상 기준과 칭호 규칙은 `docs/GAME_RULES.md`를 따른다.

### 레버리지

- 참가자 순자산을 기준으로 최대 50%까지 사용할 수 있다.
- 일반 자금과 레버리지 원금은 로트 단위로 분리해 계산한다.
- 매도 시 레버리지로 매수한 부분의 매도 시점 평가액에서 5%를 차감하고 원금을 상환한다.
- 부분 매도는 해당 수량의 레버리지 원금과 평가액을 비례 배분한다.
- 레버리지 포지션은 두 라운드 보유 후 다음 정산에서 남은 수량을 자동 청산한다.

### 일일 뉴스와 가격

- 각 종목의 뉴스와 가격은 하루 한 번만 갱신한다.
- AI는 종목 설명, 테마, 주차별 이야기, 이전 뉴스와 가격 흐름을 참고한다.
- 일일 등락률은 -30%에서 +30% 사이여야 한다. 평소에는 실제 시장처럼 작은 변동이 대부분이고 큰 변동은 강한 사건이 있을 때만 드물게 발생하게 한다.
- AI 출력은 구조화해 검증하고, 최종 가격은 서버의 규칙 기반 계산과 범위 제한을 거친다.
- 같은 실행 키를 재시도해도 뉴스, 체결, 원장 기록이 중복 생성되지 않게 한다.

## 4. 인증과 개인정보

- 카카오 로그인은 Supabase Auth를 사용한다.
- 카카오 동의 범위에서 이메일을 요청하지 않는다. 현재 프런트 범위인 `profile_nickname profile_image`를 임의로 확대하지 않는다.
- OAuth 복귀 주소는 현재 origin의 `/auth/callback`을 사용한다.
- 로그인 계정 식별은 `auth.users`, 리그 참가 데이터는 `public.randoland_participants`에 분리해 저장한다.
- 권한 판정에 사용자가 바꿀 수 있는 `user_metadata`를 사용하지 않는다.

## 5. Supabase 안전 원칙

- 대상은 DeluTive 조직의 `LCK Wiki` 프로젝트다. 원격 변경 전 대상 프로젝트를 다시 확인한다.
- 기존 LCK Wiki 테이블, 함수, 정책, 데이터는 절대 수정하거나 란도랜드2 용도로 재사용하지 않는다.
- 란도랜드2 공개 객체는 `randoland_` 접두사를 사용하고, 비공개 계산 객체는 `randoland_private` 스키마에 둔다.
- 공개 스키마의 테이블에는 RLS를 활성화하고 소유권 조건이 포함된 최소 권한 정책을 적용한다.
- `TO authenticated`만으로 사용자 데이터 접근을 허용하지 않는다. 사용자별 소유권 조건을 함께 둔다.
- 권한 오류를 피하려고 임의로 `SECURITY DEFINER`를 붙이지 않는다. 꼭 필요한 경우 비공개 스키마, 호출자 검사, 명시적 실행 권한 제한을 함께 적용한다.
- 기존에 원격 적용된 마이그레이션 파일은 수정하지 않는다. 변경마다 새 타임스탬프 마이그레이션을 만든다.
- 새 파일은 Supabase CLI 도움말과 현재 연결 상태를 확인한 뒤 `supabase migration new`로 생성한다.
- 사용자 지시에 따라 `supabase/migrations/`, `supabase/tests/local/`, `supabase/schema*.sql`, `supabase/seed.sql`은 로컬 전용이며 GitHub에 올리지 않는다.
- 스키마 변경 전후로 Advisor와 실제 테스트 쿼리를 확인한다.
- 데이터 저장 위치의 자세한 내용은 `docs/DATA_MODEL.md`, 변경 순서는 `docs/SUPABASE_WORKFLOW.md`를 따른다.

## 6. 비밀 값과 외부 API

- 브라우저에는 `VITE_SUPABASE_URL`과 publishable key만 둔다.
- `service_role`, secret key, OpenAI API 키를 `VITE_*`, 소스 코드, 로그, 문서, Git 이력에 넣지 않는다.
- `OPENAI_API_KEY`와 필요 시 `OPENAI_MODEL`은 Supabase Edge Function 비밀 값으로만 관리한다.
- OpenAI 호출은 `supabase/functions/randoland-daily-settlement/`의 서버 함수에서만 수행한다. 브라우저에서 직접 호출하지 않는다.
- 실제 환경변수는 `.env.local`에만 두고 `.env.example`에는 이름과 예시 형식만 유지한다.

## 7. 디자인 기준

- PC와 모바일은 단순 축소 관계가 아니라 각각 별도 정보 구조로 설계하고 검수한다.
- 모던한 실제 주식 거래 서비스의 인상을 유지한다.
- 기본 배경은 검은색 계열, 주요 포인트 컬러는 `#00FFBF`다.
- 한국 시장 관례에 따라 상승은 빨간색, 하락은 파란색으로 표시한다. 색상만 의존하지 말고 부호와 아이콘을 함께 쓴다.
- 모든 종목 차트는 서버 캔들 데이터 기반 캔들그래프로 표시한다.
- 종목명, 가격, 수익률, 설명, 뉴스가 깨지거나 부자연스럽게 줄바꿈되지 않게 한다. 중요한 숫자는 한 줄로 유지한다.
- 한국어 본문에는 `word-break: keep-all`을 우선하고 핵심 설명을 말줄임표로 숨기지 않는다.
- 애니메이션은 상태 이해를 돕는 150~250ms 수준으로 절제하고 `prefers-reduced-motion`을 지원한다.
- 로그인 비주얼, 종목 심볼, 빈 상태와 칭호 그래픽은 일관된 프로젝트 전용 에셋을 사용한다.
- 최소 시각 검수 크기는 PC 1440x900·1920x1080, 모바일 360x800·390x844·430x932다.
- 전체 디자인 기준은 `docs/DESIGN_SPEC.md`를 따른다.

## 8. 구현과 검증

- 현재 프런트엔드는 Vite, React, TypeScript 기반 SPA다. 기존 구조와 잠금 파일을 유지한다.
- 직접 경로와 OAuth 복귀 경로가 Vercel에서 열리도록 `vercel.json`의 SPA rewrite를 유지한다.
- 사용자 상태, 주문, 자산과 순위는 Supabase의 실제 응답을 단일 기준으로 사용한다.
- 금액과 권한을 결정하는 계산은 브라우저 값에 의존하지 말고 데이터베이스 함수 또는 서버 함수에서 검증한다.
- 주문·정산·게임 결과의 RP 증감은 자산 원장과 함께 원자적으로 기록한다.
- 여러 TSX 파일을 수정하면 반응형, 접근성, 훅 의존성, 불필요한 재렌더링을 함께 점검한다.
- 완료 전 최소 검증:
  - `npm run lint`
  - `npm run build`
- 거래나 데이터 변경은 성공 경로뿐 아니라 중복 요청, 권한 거부, 잔액 부족, 취소, 정산 잠금과 빈 상태도 검증한다.

## 9. 문서 역할

- `README.md`: 참가자용 공개 소개
- `AGENTS.md`: 현재 개발·운영 작업 지침
- `docs/GAME_RULES.md`: 상세 게임 규칙의 단일 참조
- `docs/DESIGN_SPEC.md`: 화면 설계 및 시각 QA 기준
- `docs/SUPABASE_WORKFLOW.md`: Supabase 변경 절차
- `docs/DATA_MODEL.md`: 데이터 저장 위치와 공개 범위
- `docs/VERCEL_DEPLOY.md`: 배포 환경변수와 로그인 주소 설정

규칙이나 운영 방식이 바뀌면 실제 구현과 함께 관련 문서를 같은 작업에서 갱신한다.
